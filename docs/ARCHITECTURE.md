# 🏗️ Architecture — FreeAstroAPI

## Overview

FreeAstroAPI is an astrology REST API built with **PHP 8.4 + Symfony**, served by **FrankenPHP** (Caddy + PHP worker mode) and orchestrated via **Docker Compose**.

The architecture follows the **Hexagonal (Ports & Adapters)** pattern with clear separation between domain, application and infrastructure.

---

## Layer Structure

```
src/
├── Domain/          ← Business core (zero external dependencies)
│   ├── Entity/      ← Entities: User, Report, Zodiac, Session
│   ├── ValueObjects/← Token (immutable)
│   ├── Types/       ← Enums: ReportStatus, GrantTypeLogin, LifeTimeToken
│   └── Exceptions/  ← Typed domain exceptions
│
├── App/             ← Application layer (orchestrates the domain)
│   ├── UseCase/     ← Use cases with Input/Output/Validation
│   └── Contracts/   ← Interfaces (ports) for repositories, gateways, etc.
│
├── Infra/           ← Concrete implementations (adapters)
│   ├── Adapters/    ← Database, Queue, Gateway, Encoder, Memory, Repository
│   ├── Ports/       ← Entry points: HTTP Controllers + CLI Commands
│   ├── Mappers/     ← Entity-to-database mapping
│   └── Security/    ← JWT authentication listener
│
└── Kernel.php
```

### Dependency Flow

```
Infra (Adapters) ──▶ App (Use Cases + Contracts) ──▶ Domain (Entities + Rules)
```

Dependencies always point inward. The Domain layer has no knowledge of anything external.

## Bounded Contexts

### 1. Authentication
Handles login (email/password) and JWT token refresh.

| Item | Detail |
|---|---|
| Entity | Session |
| Value Object | Token |
| Use Case | AuthenticateUseCase |
| Endpoint | `POST /api/v1/token` |
| Infra | JWT (firebase/php-jwt), Redis (sessions) |

Flow: grant_type `token` → validates credentials → generates access + refresh token → stores session in Redis.
Flow: grant_type `refresh_token` → validates refresh → generates new tokens.

### 2. User
User CRUD with automatic zodiac sign association.

| Item | Detail |
|---|---|
| Entity | User |
| Use Cases | CreateUser, FindUser, DeleteUser |
| Endpoints | `POST /api/v1/user`, `GET /api/v1/user/{id}`, `DELETE /api/v1/user/{id}` |
| Rules | Encrypted password (bcrypt), sign calculated from birth date |

### 3. Report (Astrological Report)
Most complex context — involves asynchronous processing with a state machine.

| Item | Detail |
|---|---|
| Entity | Report |
| Use Cases | CreateReportUseCase, GenerateReportUseCase |
| Endpoint | `POST /api/v1/report` |
| Consumer | `php bin/console app:consumer:report` |
| Infra | RabbitMQ (queue), MongoDB (logs), Html2Pdf (PDF generation) |

**State machine:**
```
PENDING ──▶ PROCESSING ──▶ COMPLETED
                │
                └──▶ FAILURE
```

**Async flow:**
```
HTTP Request ──▶ CreateReportUseCase ──▶ MySQL (status: PENDING)
                        │
                        ▼
                   RabbitMQ (queue: horoscope.monthly.report)
                        │
                        ▼
              GenerateReportConsumer ──▶ GenerateReportUseCase
                        │
                        ├──▶ MySQL (status: PROCESSING → COMPLETED)
                        ├──▶ PDF generated at /public/reports/
                        └──▶ MongoDB (processing log)
```

### 4. Zodiac (Support)
Support context — provides data for the 12 zodiac signs.

| Item | Detail |
|---|---|
| Entity | Zodiac |
| Repository | ZodiacRepository |
| Usage | Associated with User on creation (finds sign by birth date) |

### 5. Fortune (Support)
Provides random mystical messages for reports.

| Item | Detail |
|---|---|
| Interface | FortuneProviderInterface |
| Adapter | JsonFortuneProvider (reads from `local/fortune-message.json`) |
| Usage | Content for the PDF generated by Report |

## FrankenPHP — Worker Mode

The project uses the `dunglas/frankenphp:1-php8.4` image with Caddy configured in **worker mode**:

```caddyfile
:8000 {
    root * /var/www/html/public
    encode gzip zstd
    php_server {
        worker {
            file ./public/index.php
            num 4
            env FRANKENPHP_MAX_REQUESTS 5000
            max_consecutive_failures 5
        }
    }
    file_server
}
```

- 4 persistent workers running `public/index.php`
- Each worker recycles after 5000 requests
- gzip/zstd compression enabled
- No Nginx, no PHP-FPM — PHP runs embedded in Caddy

## Security

- **JWT (HS256)** — access token (15 min) + refresh token (7 days)
- **Redis sessions** — refresh token validated against stored session
- **Authentication listener** — routes with `_authenticated: true` are protected via AuthenticateTokenListener (Symfony Kernel Event)
- **Bcrypt** — passwords encrypted with BcryptPasswordEncoder
